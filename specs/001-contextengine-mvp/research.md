# Phase 0 Research: AISAT-INTEL MVP (Phase 1)

**Date**: 2026-06-06 | **Plan**: [plan.md](./plan.md)

All Technical Context items are resolved; the spec carries no remaining `NEEDS CLARIFICATION` markers (five were resolved in the Clarifications session on 2026-06-05). This document records the load-bearing technical decisions, their rationale, and rejected alternatives.

## 1. Access-control enforcement (data layer, not prompt)

- **Decision**: Enforce isolation at two layers: PostgreSQL Row-Level Security (`workspace_id = current_setting('app.workspace_id')`) on every tenant-scoped table, and Qdrant payload pre-filters (`workspace_id` match + `access_level Range(lte=user_access_level)`) injected before every vector search. Clearance is a fixed ladder of 5 ordered levels (1–5); a document with no explicit level defaults to the uploader's own clearance.
- **Rationale**: SC-001 is a hard release blocker (100% correctness). Prompt-level instructions are bypassable by injection; data-layer filters are not. Two independent layers give defense in depth — RLS protects relational reads, payload filters protect vector reads.
- **Alternatives considered**: (a) Prompt-only scoping — rejected: trivially defeated and untestable. (b) One Qdrant collection per workspace — rejected: collection sprawl and cost; payload isolation with indexed fields scales better for Phase 1. (c) Application-layer filtering after retrieval — rejected: leaks restricted docs into process memory and risks accidental inclusion.

## 2. Cross-clearance semantic cache keying

- **Decision**: Cache key = `sha256(workspace_id | user_id | effective_access_level | model | normalize(query))`. Personal-knowledge answers cache per-user; workspace-only answers MAY cache per `(workspace_id, access_level)` via a `cacheable_scope` flag the retriever sets when no personal document contributed.
- **Rationale**: The same query string legitimately resolves to different documents per clearance, so a query-only key would leak a higher-clearance answer to a lower-clearance member (violates SC-001). Binding tenant + identity + clearance makes a cache hit reusable only within an identical authorization scope (Clarification Q2).
- **Alternatives considered**: (a) Query-only key — rejected: cross-clearance leak. (b) No caching — rejected: needlessly sacrifices latency/cost; safe caching is achievable. (c) Per-user only for everything — rejected: lower hit rate; workspace-only answers can safely share.

## 3. Credit accounting: Redis hot path + Postgres ledger

- **Decision**: Redis holds the authoritative hot balance (`DECRBY`, atomic); PostgreSQL `credit_ledger` (append-only, `idem_key UNIQUE`) is the durable source of truth, written async via a Redis outbox drained by the **Go kernel billing worker — the sole `credit_ledger` writer** (`backend-go/kernel/billing/`). Python spend producers (query/ingest/agent workers) compute cost and **publish** `billing.deduct.<ws>` events but never write the ledger or touch the Redis balance, keeping one authoritative money-writer behind the kernel and RLS. Every credit-affecting operation carries an idempotency key; a `SET NX billing:applied:{idem_key}` guard plus the ledger unique index make retries/double-clicks no-ops. Cold-start rehydration rebuilds the Redis balance from the ledger under a per-workspace lock; an hourly reconciliation cron corrects drift with `operation_type='reconcile'` rows. The outbox key space and the reconciliation scan are **partitioned by workspace shard** from day one: Phase 1 runs a single drainer + single hourly reconcile, but the shard key lets Phase 4 run N parallel drainers/reconcilers without redesigning the key space or risking double-apply (§14). The drainer and reconcile loops run in the dedicated `cmd/worker` role (external CronJob → NATS tick → queue-group consumer, idempotent atomic claims), never as in-process timers in the request-serving `cmd/api` tier (§15).
- **Rationale**: SC-006 demands exact accounting and no double-charge. Atomic Redis decrement gives sub-ms enforcement on the hot path; the durable ledger guarantees recoverability and auditability; idempotency closes the retry double-charge hole.
- **Alternatives considered**: (a) Postgres-only synchronous deduction — rejected: too slow on the hot path, lock contention. (b) Redis-only — rejected: not durable; a Redis loss is a billing incident. (c) Best-effort ledger write — rejected: drops under failure; the outbox makes deduction and ledger intent atomic.

## 4. Provider fallback strategy (one-hop)

- **Decision**: LLM aliases (`fast`, `smart`, `embed`, `rerank`) resolve to `{primary, fallback}` in the Python `llm_gateway`. Fail over on timeout / 5xx / rate-limit only, capped at one hop, with an `llm.fallback.count` metric and a circuit breaker. `embed` has **no per-call fallback**: a down primary embedder parks the chunk in `ingestion.dlq` for retry rather than embedding with a different model.
- **Rationale**: Edge case + FR-029 require degradation, not outage, while never failing over on "low-quality" output (that is an eval concern). Embedding models cannot be mixed within one Qdrant collection without making distances meaningless, so per-call embed fallback would silently corrupt retrieval quality.
- **Alternatives considered**: (a) Multi-hop fallback — rejected: unbounded latency. (b) Quality-based fallback — rejected: undefined/abusable signal, not a reliability event. (c) Per-call embed fallback — rejected: index inconsistency.

## 5. Prompt-injection structural defenses (Phase 1, not deferred)

- **Decision**: Ship the structural defenses in Phase 1: (1) retrieved content wrapped in `<retrieved_document>` delimiters with a system rule that delimited text is data, never commands; (2) tool results never trigger new tool calls without the router re-deriving intent from the original classified intent; (3) strict per-role `allowed_tools` allowlist enforced on every dispatch; (4) read-only-only Phase 1 toolset (no write/send tools); (5) audit + `result_hash` on every tool call. A Node 0 moderation gate short-circuits disallowed/injection inputs before any retrieval or spend.
- **Rationale**: Uploaded files and crawled pages are untrusted content on the core data path — the textbook injection vector. These defenses are cheap, are not a scanning tool, and are required for FR-010/FR-011/FR-012 and SC-007. Automated red-teaming (Garak) is Phase 3 and layers on top, not a replacement.
- **Alternatives considered**: Deferring all injection handling to Phase 3 — rejected: leaves the core data path exposed during Phase 1.

## 6. Async agent execution over NATS (not in-process)

- **Decision**: The Go BFF publishes queries to `query.agent.<workspace_id>` and returns an SSE stream ID; a Python worker pool consumes via a **NATS JetStream durable pull consumer bound to a per-subject queue group** (not core NATS), runs the LangGraph 7-node graph, checkpoints state to Redis (AOF), and streams partial results via Redis pub/sub → Go SSE → browser. Only `intent=long_horizon` runs create a durable `agent_run` row; short interactive queries do not. The Go SSE relay is built behind a clean boundary (it only subscribes to Redis pub/sub by `stream_id` and forwards) so it is a **logically separable tier** from the request-handling BFF (see §14).
- **Rationale**: Decouples request latency from model latency, enables independent worker scaling per NATS subject via the queue group (KEDA on JetStream consumer lag in Phase 2), and survives worker restarts via JetStream redelivery + Redis checkpoints. Core NATS would lose in-flight messages on a worker crash and expose no lag metric to autoscale on; JetStream is therefore a Phase 1 lock-in (§14). Matches the streaming UX (SSE event taxonomy) and the durability needs of long-horizon tasks (SC-009).
- **Alternatives considered**: (a) In-process LangGraph per HTTP request — rejected: couples latency, no durability, poor scaling. (b) WebSockets — rejected: SSE is sufficient for server→client streaming and simpler behind CDN. (c) Core (non-JetStream) NATS — rejected: no persistence, no durable consumers, no consumer-lag signal — unfixable later without rewriting every publisher/consumer.
- **Why two transports (JetStream + Redis pub/sub), not one**: the two hops have opposite needs. Hop 1 (BFF→worker) is *durable work dispatch* — must survive a crash, be redeliverable, and expose consumer lag → JetStream. Hop 2 (worker→relay→browser) is *ephemeral live-token fan-out* to an already-connected client — lowest latency, simple per-`stream_id` fan-out, fire-and-forget → Redis pub/sub (already the streaming control plane: checkpoints, cache, credits). Using JetStream for hop 2 would add per-token persistence + per-query stream/consumer churn for a durability nobody needs (a lost *token* is cosmetic; credits/audit/final answer are authoritative in Postgres). Crash recovery already lives at the right layer: JetStream redelivers the *job*, the Redis checkpoint resumes the *run*, the relay just re-subscribes. If replay-on-reconnect is ever required, the upgrade is a **Redis Stream** (`XADD`/`XREAD`) per `stream_id` — same Redis, not JetStream.

## 7. Structured (Tier 2) data access: fixed tools, not Text-to-SQL

- **Decision**: Structured-data questions are answered by fixed, parameterized, workspace-scoped MCP tools (`query_employees`, `query_projects`, `query_metrics`). The LLM chooses a tool and supplies typed arguments; the SQL is hand-written and scoped — never free-form generated SQL.
- **Rationale**: FR-008 + SC-001. Free-form generated SQL is an injection and data-exfiltration risk and cannot be reliably scoped to a workspace/clearance. Fixed tools keep the access boundary in code.
- **Alternatives considered**: Free-form Text-to-SQL — rejected: unbounded query surface, unsafe across tenants.

## 8. Ingestion pipeline + upload boundary

- **Decision**: Browser uploads directly to S3 via presigned URL; an S3 event publishes to `ingestion.{mime}.{workspace_id}`; Python workers route by MIME (MarkItDown for PDF/DOCX/MD, GPT-4o-mini captioning for images; video/audio is a registered-but-501 stub). Web links are no longer a standalone ingestion path — they are crawled (Crawl4AI, via the shared `web_distill` capability) inside member-initiated note enrichment, and only the member-accepted note body is ingested (FR-001). BAML extracts advisory metadata (tags/data_type/summary/suggested_sensitivity); security fields are stamped server-side. A per-file size limit (admin-configurable per workspace, default 50 MB) is enforced at the upload boundary before any ingestion or spend.
- **Rationale**: Direct-to-S3 keeps large payloads off the app servers; MIME routing matches FR-001/FR-002/FR-003; the size limit (Clarification Q4) bounds cost and processing time and gives a clear oversize rejection rather than silent failure.
- **Alternatives considered**: (a) Proxying uploads through the BFF — rejected: memory/bandwidth pressure. (b) Model-inferred access level — rejected: a malicious upload could self-tag permissive (FR-004/FR-005).

## 9. PII scrubbing & retention window

- **Decision**: A PII filter runs in the gateway **before** any prompt/response is written to a trace or eval store. Raw prompt/response bodies are retained for a **30-day** window, after which they are purged (partition `DROP`) and only metadata, hashes, and aggregates remain. `llm_call_log` and `agent_audit_log` never store raw message bodies — only metadata, token counts, and hashes.
- **Rationale**: FR-024 + Clarification Q5 fix the window at 30 days. Scrub-before-write minimizes PII exposure; partition-drop expiry is cheap and tombstone-free. (Note: the source draft mentioned a 7-day Langfuse window; the clarified spec value of 30 days is authoritative and supersedes it.)
- **Alternatives considered**: (a) Indefinite raw retention — rejected: privacy/compliance risk. (b) Scrub-after-store — rejected: raw PII briefly persists. (c) Row-delete expiry — rejected: expensive, leaves tombstones.

## 10. Redis role separation (single cluster, logical split)

- **Decision**: Phase 1 runs one Redis cluster but separates roles by logical DB + key-prefix: durable state (credit balance, idempotency keys, `agent_run` checkpoints) runs `noeviction` + AOF; ephemeral cache (semantic/exact cache, hot-index flags) runs `allkeys-lru`, AOF off; counters (rate limits, daily budgets) run `volatile-ttl`. Pub/Sub for SSE fan-out is transient.
- **Rationale**: These roles have incompatible durability/eviction profiles — an LRU policy correct for the cache would silently evict credit balances or in-flight checkpoints (a billing/data incident). Logical separation makes the Phase 2 split into independent clusters a config change, not a refactor.
- **Behavior under Cluster / failover** (why scaling Redis is safe): a `SET NX` lock is **not** a correctness primitive under async-replication failover — a primary can ACK the lock, die before replicating, and a promoted replica grant it twice (the standard Redlock critique). The design never relies on it: every Redis guard has a Postgres backstop (`credit_ledger.idem_key UNIQUE`, notifications `(user_id, idem_key) UNIQUE`, the janitor's conditional `UPDATE … WHERE status='running'` fencing claim — §15), so a two-leader failover window produces zero duplicate effects. Three residual *scale* (not correctness) requirements when moving to Redis Cluster: (a) **hash-tag a workspace's keys** (`credit:{ws}:balance`, `billing:applied:{ws}:{idem}`) so they colocate on one slot and stay atomically scriptable across slots; (b) the hot `DECRBY` balance is a **cache** — a lost decrement on failover is a temporary-overspend/RPO question corrected by the hourly reconcile, never a double-charge; (c) rate-limit/budget counters **and** the opaque session store are Redis-only and **fail-safe** (a failover lets a few extra ops through / logs users out — best-effort, not a money or security hole). Verifying reconciliation + rehydration across failover is [phase-4 item 6](../draft-plan.md#6-redis-high-availability).
- **Alternatives considered**: One Redis with a single eviction policy — rejected: cache pressure would evict durable state.

## 11. Local agents (optional, additive)

- **Decision**: Local agents are optional; all core features work with zero agents (SC-008). A registered device gets a scoped, revocable PAT (user + workspace). Default `proxy` sub-mode routes LLM calls through `/llm/proxy` (metered, moderated, audited); `byok` sub-mode bypasses server LLM metering/moderation but still routes MCP tool calls server-side and is disableable per workspace by an admin. All agent results are untrusted; tool arguments are validated against the PAT scope. Long-horizon runs are durable with a hard per-run `credits_cap`.
- **Rationale**: FR-025–FR-028 + SC-008/SC-009. Optionality keeps the core product independent of agents; the proxy default preserves metering/audit; BYOK is an explicit, admin-gateable trade-off.
- **Alternatives considered**: Making agents mandatory for any feature — rejected: violates SC-008.

## 12. Context compression (Headroom) — deferred to Phase 2, seam in Phase 1

- **Decision**: Do **not** integrate a context-compression layer ([Headroom](https://github.com/chopratejas/headroom)) in Phase 1. Instead, treat `services/llm_gateway.py` as the single, documented seam where a pre-send compression transform (and post-retrieval chunk compression) could later slot in without changing the gateway's public interface or any caller. Phase 2 evaluates Headroom in **library mode inside the existing gateway** (not proxy mode — the gateway already owns aliases, fallback, budget, and semantic cache; a second proxy would be redundant), behind a kernel `Flags` gate, with `compress` on and the output/effort shaper **off**.
- **Rationale**: The gateway chokepoint (§4) and per-token credit metering (§3) make token reduction land in exactly one place and map directly to lower spend per answer — a strong fit. But three release-blocking constraints make it a *measured*, not assumed, change: (a) **citations** (SC-002/SC-003) — compressing retrieved chunks before the LLM can degrade citation fidelity, so it must clear the `evals/run.py` gate (recall@k, MRR, citation accuracy) first; (b) **access control = 100% correctness, release blocker** (SC-001) — Headroom's reversible cache (CCR) persists plaintext originals of already-retrieved, access-scoped content locally, a *new* sensitive datastore that would need the same RLS/tenant-isolation guarantees as Postgres/Qdrant or it becomes a leak path (FR-030 access-filter assertion must extend to it); (c) **determinism / refuse-before-spend** (SC-007) — the output shaper mutates model behavior and the system prompt, which fights the deterministic, contract-first posture, so it stays off initially. Operationally it also drags in a Rust toolchain, ONNX runtime, and an HF model download — heavy for the MVP's clean kernel/product split.
- **Alternatives considered**: (1) Headroom **proxy mode** in front of providers — rejected: duplicates the gateway's existing fallback/cache/metering chokepoint and adds a second moderation/metering surface. (2) Integrating compression in Phase 1 — rejected: unverified citation/access-control impact against release-blocking SCs. (3) Building a bespoke compressor — rejected for Phase 1 scope; revisit only if Headroom's eval results are insufficient.

## 13. Agent memory (Mem0) access-control scoping

- **Decision**: Treat the Mem0 memory store (graph Node 5) as a **third access-controlled surface** alongside Qdrant retrieval and the semantic cache. Every memory is written with `workspace_id`, `user_id`, and an `access_level` stamp equal to the **highest `access_level` among the chunks/answer that produced it** (a memory is only as shareable as its most-restricted source). Node 5 injection applies the same predicate as retrieval — `inject WHERE workspace_id == ctx AND user_id == ctx AND access_level <= effective_access_level` — evaluated against the requester's **current** clearance at read time. A memory derived from a doc above the requester's current clearance is silently withheld, never injected.
- **Rationale**: SC-001 forbids an answer being *influenced by* a document above clearance. Without a stamp+filter, two paths leak: (a) **temporal demotion** — a member lowered L4→L2 (the `clearance_changed` event) still carries L4-derived facts in personal memory; (b) **unfiltered injection** — Node 5 currently injects by `user_id` only, with no clearance gate. The semantic cache already binds `effective_access_level` into its key (§2); memory is the same surface and must get the same treatment, or it becomes the one unguarded path around RLS + payload filters. Memory is personal (`user_id`-scoped), so injected-memory answers are `cacheable_scope=per-user` only, and the cache key is computed **after** memory filtering — keeping the cache invariant intact.
- **Alternatives considered**: (a) No clearance stamp on memory (status quo) — rejected: direct SC-001 leak via demotion and cross-context bleed. (b) Hard-delete memories on demotion — rejected: lossy and irreversible (a re-promotion can't restore them); a read-time filter is reversible and cheaper. (c) Disable memory entirely — rejected: needlessly drops the FR-009 coherent-follow-up capability when a stamp+filter is sufficient.

## 14. Scale-forward seams locked in Phase 1 (rework-risk decisions)

These five decisions are fixed in Phase 1 not because Phase 1 needs the scale, but because deferring them past implementation turns an additive Phase 4 change into a rewrite. They cost nothing now and preserve the option to scale later. Provisioning the scale itself — KEDA, PgBouncer, Redis/Qdrant HA, load tests — remains Phase 4 ([draft-plan.md — Phase 4](../draft-plan.md#phase-4-scalability-and-resilience-hardening)).

- **NATS = JetStream, durable pull consumers, per-subject queue groups (not core NATS).** Crash recovery, DLQ redelivery, and autoscale-on-consumer-lag all assume persistent streams + durable consumers. Core NATS would lose in-flight messages on a worker crash and leave no lag metric to scale on — unfixable without rewriting every publisher/consumer. Queue groups make adding pods pure horizontal scale-out. (§6, [contracts/nats-subjects.md](./contracts/nats-subjects.md))
- **The SSE/edge relay is a logically separable tier from the request-handling BFF.** REST aggregation (CPU-bound, short-lived) and SSE termination (connection-bound, long-lived) have opposite scaling profiles. Packaged like the Python tier (one image, role-selected at deploy): the Go BFF is a **single image with two entrypoints** — `cmd/api/main.go` (REST) and `cmd/relay/main.go` (streaming) — sharing all `internal/` code, auth, the Redis pub/sub client, and the SSE event contract, so the `stream_id` producer (`POST /query`) and consumer (`GET /query/{streamId}`) can never version-skew. Phase 1 MAY co-deploy them in one process; Phase 4 deploys `api` and `sse-relay` as independently-scaled services. Stream *creation* (`POST`) is an `api` concern; only the long-lived `GET` belongs to the relay. Because the relay subscribes to Redis pub/sub keyed by `stream_id` (not in-process worker output), **any relay replica can serve any stream — no sticky sessions** — making it trivially horizontally scalable. (§6, [contracts/sse-events.md](./contracts/sse-events.md))
- **The Redis credit outbox key space is workspace-partitionable from day one.** Phase 1 runs a single billing-worker drainer + single hourly reconciliation, but outbox keys and the reconcile scan are sharded by workspace so Phase 4 can run N parallel drainers/reconcilers by shard without redesigning the key space or risking double-apply. (§3)
- **Qdrant stays payload-isolated within shared `personal`/`workspace` collections, with a documented re-shard/replication trigger.** The shared-collection + payload-filter choice (§1) is compatible with Qdrant's within-collection sharding and replication (additive). The watch-item is collection size: a re-shard / cluster-replication decision is triggered when the `workspace` collection approaches its node memory/latency budget — flagged now as a planned threshold, not an incident. (§1)
- **Scheduled & background work runs single-owner in a dedicated `cmd/worker` role, never inside a request-serving tier.** Periodic jobs (outbox drain, hourly reconcile, stale-heartbeat janitor, DLQ sweeps, `llm_cost_daily` matview refresh) are triggered by an external CronJob → NATS tick → JetStream queue-group consumer, and every action is an idempotent atomic claim — so adding `api`/`relay`/`worker` replicas can never double-fire a timer. Embedding these loops in `cmd/api` now would make Phase 4 autoscaling silently multiply them (N concurrent reconcilers/janitors); extracting them later means re-homing every job. (§15)

## 15. Scheduled & background work — single-owner execution (no duplicate timers)

- **Decision**: No scheduled or background loop runs inside a request-serving tier. All periodic/background work — credit-outbox drain, hourly reconciliation, `agent_run` stale-heartbeat janitor, DLQ retry sweeps, and the `llm_cost_daily` matview refresh — runs in a dedicated **`cmd/worker` role** (same Go image as `cmd/api`/`cmd/relay`, role-selected at deploy; §14), with the Python stale-heartbeat janitor likewise deployed as its own single-owner role rather than a loop inside every `long_horizon` pod. Scheduling is **external and event-driven** — a pluggable scheduler (Kubernetes `CronJob`, DigitalOcean App Platform scheduled component / Function, or a plain `cron`/systemd timer on a Droplet; locally a Make target or a single in-process ticker in the one `worker`) emits a NATS JetStream tick (`billing.reconcile.tick`, `agent.janitor.tick`, `usage.matview.refresh`). The “no in-process `time.Ticker`” rule applies to the **request-serving tiers** (`api`/`relay`); the single-owner `worker` MAY use an internal ticker, since correctness rests on the atomic claim below, not on the trigger. Each tick / outbox / DLQ subject is consumed by a **durable queue group → exactly one worker handles each event**, with JetStream redelivery on crash. Correctness never depends on “only one replica is running”: every scheduled action is an **idempotent atomic claim**, so a duplicate delivery is a no-op — the janitor re-queues only via a conditional transition (`UPDATE agent_run SET status='queued' WHERE id=$1 AND status='running' AND last_heartbeat_at < $2 RETURNING id`), reconcile guards each hour-bucket with `SET NX reconcile:lock:{shard}:{hour_bucket}`, and the outbox uses an atomic pop (`LPOP` / Stream consumer-group) plus `credit_ledger.idem_key UNIQUE`. The matview refresh uses `REFRESH MATERIALIZED VIEW CONCURRENTLY` (requires a UNIQUE index) so a duplicate tick is at worst redundant work, never a lock error.
- **Rationale**: Background loops embedded in the `api` tier multiply with replicas (2 today, autoscaled in Phase 4), so a “single hourly reconcile” silently becomes N concurrent reconcilers — racing `reconcile` deltas (SC-006), triple-re-queued agent runs and triple spend (SC-009), and colliding `REFRESH MATERIALIZED VIEW`. Separating *who schedules* (CronJob), *who executes* (queue-group worker), and *what guarantees correctness* (atomic claim + data-layer idempotency) makes the common path need **no leader election**, and keeps Phase 4 horizontal scaling purely additive (more workers = more throughput; the §3 shard keys keep ownership clean). Leases and queue groups are throughput optimizations; the data-layer idempotency is the **only** correctness guarantee — so a transient two-leader window (deploy overlap, network partition, lease handoff) still produces zero duplicate effects.
- **Alternatives considered**: (a) In-process `time.Ticker` in `cmd/api` guarded by a Redis leader lease — rejected: a lease held by a crashed pod, or handoff during a rolling deploy, transiently yields two leaders, and correctness would rest on that weakest link. (b) A durable-execution engine (Temporal / River / Asynq) — rejected for Phase 1: JetStream redelivery + Redis checkpoints + atomic claims already give durable, exactly-once-*effect* execution; a workflow engine is a Phase 4+ “if long-horizon orchestration outgrows LangGraph” decision, not an MVP dependency. (c) Postgres `pg_try_advisory_lock` as the primary guard — kept as the **optional HA gate** for any residual in-process loop (auto-released on connection drop, unlike a TTL lease) or the k8s `Lease` API, but not the primary mechanism, which stays the external trigger + idempotent claim.

## 16. Chunking strategy — structure-aware + parent/child + contextual retrieval

- **Decision**: Three layered techniques in `services/ingestion/chunker.py`. (1) **Structure-aware splitting** — split on document structure (markdown headings, paragraph/sentence boundaries, table rows, code fences) rather than a fixed character window, so a chunk never straddles a heading or cuts mid-sentence; fall back to a recursive token splitter only for unstructured text. (2) **Parent/child ("small-to-big")** — the *child* chunk (~200 tokens) is what gets embedded and searched (precise match); on a hit, retrieval expands to its *parent* (~1000 tokens, linked by `parent_doc_id`) before sending to the LLM (complete context for grounded citation). (3) **Contextual retrieval** (Anthropic-style) — before embedding, each child chunk is prefixed with a short LLM-generated situating line ("This excerpt is from *<doc title>*, section *<…>*; it discusses *<…>*") produced via the `fast` alias, with the per-document summary computed **once** and reused across that document's chunks to bound cost. The prefix is embedded *with* the child (lifting recall on pronoun-/acronym-/context-poor fragments) but is **not** a citation span. The contextual-prefix step is **flag-gated** (kernel `Flags`, `chunking.contextual_prefix`), defaults on for text documents, and must clear the `evals/run.py` recall/MRR/citation gate before it is enabled in CI.
- **Rationale**: Naive fixed-window chunking is the single biggest avoidable recall loss in production RAG — context-poor fragments embed ambiguously, and a mid-sentence cut corrupts both the retrieval unit and the cited span (SC-002/SC-003). Structure-aware boundaries + small-to-big is the standard answer: a *precise* retrieval unit plus a *complete* generation unit. Contextual retrieval is a measured, high-ROI recall win (Anthropic reports a large drop in retrieval-failure rate from per-chunk situating context) that fits cleanly here because the `fast` alias + per-doc summary reuse keep the added ingestion cost bounded and the gateway already meters/traces it. Gating it behind a flag + the eval gate keeps citation fidelity a *verified* property, not an assumption.
- **Alternatives considered**: (a) Fixed-size character windows with overlap — rejected: ambiguous embeddings and broken citation spans. (b) Semantic chunking (embedding-distance boundary detection) — deferred: higher ingestion cost and tuning burden for marginal gain over structure-aware boundaries on the target corpora; revisit only if eval recall stalls. (c) Embedding the contextual prefix as a *separate* vector — rejected: doubles index size and splits a chunk's signal across two points; prefix-then-embed keeps one vector per child.

## 17. Answer groundedness & self-correction (CRAG-style) — deferred to Phase 2, graph seam in Phase 1

- **Decision**: Do **not** add an answer-grounding / self-correction loop in Phase 1. The Phase-1 graph stays deterministic and linear (moderate → rewrite → retrieve → rerank → expand → memory → generate+cite). Keep the LangGraph graph the single seam where a **groundedness gate** can later slot in as an *additive* node between rerank/expand and generate (Node 7) without changing any existing node's contract. Phase 2 adds a **CRAG-style** ("Corrective RAG") evaluator node: it grades retrieval relevance (a lightweight grader via the `fast`/`rerank` alias) and, on a low-confidence/empty result, either (a) triggers one query reformulation + re-retrieve, (b) routes to the Phase-2 `web_search` tool (per-search HITL — the note-enrichment §Phase-2 seam), or (c) **abstains** ("I don't have enough information") rather than generate from weak context. A Self-RAG-style post-generation faithfulness check (answer-vs-sources entailment) is the same node's optional second half.
- **Rationale**: The linear Phase-1 graph already clears the spec's grounded-citation bar via the eval gate (recall@k, MRR, citation accuracy) and the hard zero-above-clearance assertion; correction loops add LLM round-trips, latency, and credit cost, and an abstain path interacts with refuse-before-spend (SC-007) and the suggestion-suppression rule (FR-031) — all of which want eval evidence before shipping. Deferring keeps Phase 1 deterministic (the release-blocking posture for SC-001/SC-007) while the existing graph + the `web_search` seam make the Phase-2 addition **one new node + edges, not a refactor**. The retrieval-confidence signal a grader needs is already captured in the Phase-1 debug trace (rerank scores).
- **Alternatives considered**: (a) Ship CRAG in Phase 1 — rejected: unverified latency/cost/abstain-UX impact against release-blocking SCs, competing with the access-control and metering work that gates the MVP. (b) Always re-retrieve on every query (no grader) — rejected: doubles retrieval cost for the common already-grounded case. (c) Free-form ReAct re-planning from tool output — rejected: re-deriving next steps from (untrusted) tool/document output is exactly the prompt-injection escalation path Phase 1 closes (FR-011); any Phase-2 loop must re-derive from the original classified intent, never from tool output.

## 18. Dead-letter drain & poison-message handling — bounded re-drive, then terminal park

- **Decision**: Every DLQ subject (`ingestion.dlq.<ws>`, `notify.email.dlq.<ws>`, and any future `*.dlq.<ws>`) is drained by a single **generic DLQ sweeper** that runs single-owner in `cmd/worker`, triggered by an external `dlq.sweep.tick` (the same pluggable scheduler seam as §15) and consumed by a durable queue group → exactly one worker per tick. Each parked message carries a `dlq_attempts` counter and a `first_failed_at`. For each message the sweeper either (a) **re-drives** it — if `dlq_attempts < MAX_DLQ_ATTEMPTS` (default 5) and an exponential-backoff window keyed on `first_failed_at` has elapsed, it re-publishes the message to its **original work subject** (the embed step for `ingestion.dlq`, `notify.email.<ws>` for the email DLQ) with `dlq_attempts+1`, so the **owning tier reprocesses it** (Python re-embeds with the *original* model only, §4; the Go email worker re-sends) — or (b) **terminally parks** it — once `dlq_attempts ≥ MAX_DLQ_ATTEMPTS`, it writes the message to a durable `dead_letters` Postgres table (queryable by admins for inspection / manual replay) and emits a `dlq.dead.count` metric for alerting; the message is never re-driven again.
- **Why re-drive to the live subject, not reprocess inside the sweeper**: the sweeper is a generic Go mover that knows nothing about embedding or SMTP. Re-publishing to the original subject keeps the actual retry in the tier that owns it (Python embed / Go mailer) and reuses that worker's existing idempotency — chunk identity for embeds, and the email worker's "mark sent only after provider-accept" rule — so a re-drive of a genuinely-unsent item delivers exactly once while a duplicate is a no-op. Same separation as §15: *who schedules* (cron tick) vs *who executes* (owning worker) vs *what guarantees correctness* (data-layer idempotency).
- **Why a hard attempt cap**: without a terminal state a poison message (malformed payload, permanently-rejecting recipient) loops forever, burning provider calls and masking the failure. `MAX_DLQ_ATTEMPTS` + the `dead_letters` table converts an unbounded retry into a bounded, observable, replay-on-demand outcome — "never dropped" becomes "either recovered or visibly dead," not "retried indefinitely." For email specifically, hard bounces/complaints short-circuit even earlier via `email_suppressions` (US8), so the cap mainly catches transient-looking-but-permanent failures.
- **Alternatives considered**: (a) JetStream `MaxDeliver` + a native DLQ only (no sweeper) — rejected: gives in-stream redelivery but no *cross-outage* re-drive (a message parked after `MaxDeliver` just sits), no backoff control, and no queryable terminal store. (b) Per-pipeline bespoke retry loops — rejected: duplicates backoff/cap/poison logic in every worker and re-tier-couples the sweeper. (c) Infinite re-drive with alert-only — rejected: a poison message loops forever and the alert never clears.
