Organization

Organization roles are administrative reach, never content reach. An org admin can create workspaces, manage members, connect the identity provider and pay the bill — and cannot read a single document in any workspace without being separately granted membership there. Knowledge never crosses a workspace boundary, and this screen does not change that.

Workspaces (3)

Each workspace is an isolated knowledge boundary. Credits are allocated from the organization pool.

A workspace is a knowledge domain, not an org-chart node. Most organizations should run one workspace and separate access with clearance levels and groups — that is what those two axes are for. Create additional workspaces only for genuinely separate knowledge domains: client isolation, pre-close M&A, or data-residency requirements. A workspace per department fragments the knowledge base and agents can no longer find one authoritative answer.

Clearance scheme

Level names are display only — the stored value is the number, so renaming never re-indexes anything.

  • L5
  • L4

Renaming a level is safe. Removing one is not. Reducing the level count requires an explicit remap for every document at the removed level — otherwise those documents become unreachable, or fall to a lower level and widen access silently. The remap runs before the change commits.
